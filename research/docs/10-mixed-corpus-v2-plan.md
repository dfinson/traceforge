# 10 — Mixed Corpus v2 — Plan

## Why v2

v1 outcomes ([09-golden-corpus-v1-outcomes.md](09-golden-corpus-v1-outcomes.md))
landed honest labels on 300 sessions, but the WSL Copilot session-store is
overwhelmingly CodePlane utility-LLM calls. Rare-class supply hits 8–30% of
N_min for both classifiers. The labels are correct; the corpus is the wrong
shape. The fix is to **mix in real coding-agent trajectories from sources
where they are abundant**, not to relabel the same dataset.

## Supply diagnosis

Re-survey of the same WSL DB (`research/scripts/survey_session_store.py`):

| Filter | Sessions |
| ------ | -------: |
| Total session stubs                 | 23,299 |
| Have any turn                       | 1,523 |
| Have ≥5 turns                       |   360 |
| Have ≥1 `session_files` row         | 1,914 |
| Have ≥5 turns AND ≥1 tool file      |    49 |
| Have ≥10 turns AND ≥10 tool files   |    13 |

`session_files` only tracks `edit`/`create` (not read/grep/bash), so the
real "agent session" count in WSL is somewhat higher than 49 — v1 ingest of
349 sampled sessions found 39 with ≥1 tool event, extrapolating to ~170 in
the full population. **The WSL ceiling for agent sessions is in the low
hundreds.** Insufficient for boundary classifier alone.

## Sources for v2

| Tag | Source | Format | Expected agent sessions | Notes |
| --- | ------ | ------ | ----------------------: | ----- |
| `copilot-cli` | WSL `~/.copilot/session-store.db` re-mined | sqlite → tracemill ingest | ~150 | Already have adapter; widen the manifest filter to "any tool event" not "min turns" |
| `swe-agent-nebius` | [`nebius/SWE-agent-trajectories`](https://huggingface.co/datasets/nebius/SWE-agent-trajectories) on HF | 12-shard parquet, JSON `trajectory` per row | 600 sampled from resolved subset | Apache 2.0 / CC-BY-4.0; 13,389 resolved runs available; avg 31 steps |

Deliberately scoped to two sources for v2. A third (SWE-smith mini-agent,
Aider chat logs) is queued for v3 if v2 doesn't hit the targets.

### Why nebius

* **Real coding-agent trajectories**: agent attempts to solve a real GitHub
  issue end-to-end with shell + edit tools. The "trajectory" field is a JSON
  list of role-tagged events (ai: reasoning + actions, user: observations)
  that maps cleanly onto tracemill's event schema.
* **Volume**: 13,389 *resolved* runs available; we sample 600. Failed runs
  (66,647) are excluded for v2 — they contain useful exploration/verification
  patterns but their distribution is skewed by the agent giving up.
* **Length fits**: avg 31 steps for resolved runs, well under our
  `max_events_per_call=220` cap. We additionally filter to runs whose
  event count after enrichment is ≤200.
* **Licensing**: dataset card lists CC-BY-4.0; trajectories generated by a
  research project for explicit reuse.

### Why not other obvious candidates

* **OpenHands trajectories** — no consolidated public dump on HF.
* **SWE-smith mini-agent (Kwai-Klear)** — strong candidate but younger
  (Nov 2025) with less documentation; deferred to v3.
* **Aider chat logs** — markdown-formatted, would need a custom parser
  that's worth its own design pass; deferred.
* **Cognition Devin traces** — not publicly published.

## Target sample sizes

From [05-data-sizing.md] N_sweet targets:

| Classifier | N_sweet sessions | Rare-class rows needed | Mix to hit it |
| ---------- | ---------------: | ---------------------: | ------------- |
| Phase      | 400              | review ≥2,100          | 150 copilot-cli + 250 swe-agent (≥30-event avg) |
| Boundary   | 800              | step ≥1,750            | 150 copilot-cli + 600 swe-agent                 |

**Combined v2 manifest target: 750 sessions** (150 copilot-cli + 600
swe-agent-nebius). Budgeted to land between N_sweet and N_max for the
boundary classifier (the binding constraint), with stratified sampling
across step-count buckets to avoid loading the corpus with one trajectory
length.

## Stratification

Within `swe-agent-nebius`, stratify the 600-trajectory sample across
step-count buckets so we get coverage of short and long agent sessions:

| Bucket (steps) | Share of sample |
| -------------- | --------------: |
| 1–10           | 20% (120) |
| 11–25          | 35% (210) |
| 26–50          | 30% (180) |
| 51–100         | 12% (72) |
| 101–200        | 3% (18) |

These shares come from the dataset's published step distribution and our
constraint that very long trajectories may not fit the labeler's single-call
budget. Bucket boundaries themselves go in YAML, not hardcoded.

For `copilot-cli`, we take everything that passes the tool-event filter
after re-ingest (likely ~120–170 sessions). No stratification needed —
small enough to use entirely.

## Schema work

Add a **`source`** column everywhere a row carries a `session_id`:

* `data/interim/labeling-corpus/<source>/<sid>.parquet` — new sub-folders
  per source so paths can't collide on uuid.
* `data/interim/labeling-manifest.yaml` — each entry gets `source` and
  `bucket` (for swe-agent).
* `data/processed/labels/<sid>.json` — record `source` alongside
  `session_type`.
* `data/processed/{phase-labels,boundary-labels,activity-step-toc}.parquet`
  — new `source` column.

## Adapter contract

Each per-source ingest produces a parquet matching the tracemill enricher
output schema (one row per event, sorted by `seq`). Required columns:

* `event_id` (str, uuid)
* `seq` (int)
* `kind` (str — `message.assistant`, `tool.call.<name>`, `tool.result.<name>`)
* `ts` (float — relative or wall-clock seconds)
* `payload` (json string — tool args / message text)
* `metadata.phases` (list[str] — from enricher; may be empty for new sources
  that haven't been classified yet)

The two adapters:

1. **`copilot-cli`** — already lives in `tracemill_research/ingest/copilot.py`.
   Only change: drop the `min_turns` floor at the manifest layer; keep all
   sessions where the enricher emits ≥1 non-message event.
2. **`swe-agent-nebius`** — new module at
   `tracemill_research/ingest/swe_agent.py`. Takes a HuggingFace parquet
   shard, walks each row's `trajectory` JSON, emits one tracemill event per
   trajectory entry. Tool-name extraction follows SWE-agent's action format
   (commands like `open`, `goto`, `edit`, `bash`).

## Steps to execute

Each is a separate script under `research/scripts/`:

1. `survey_session_store.py` — done; produced the supply numbers above.
2. `download_swe_agent_shard.py` — HF dataset download with manifest of
   which shards were pulled (provenance for reproducibility).
3. `ingest_swe_agent.py` — convert a downloaded shard → per-session
   parquets under `data/interim/labeling-corpus/swe-agent-nebius/`.
4. `build_labeling_manifest.py` — extended to compose from multiple
   sources with stratified sampling.
5. `label_corpus.py` — already source-aware via the new `source` field on
   manifest entries.
6. `build_training_tables.py` — already aggregates per-row; add the
   `source` passthrough.
7. `run_pilot_eval.py` — break down counts by `source` × `session_type`.

## What "data ready" means for this task

Out of scope tonight: the labeling pass. The user explicitly asked to **plan,
document, and get the data ready**. "Ready" means:

* Plan doc committed (this file).
* Two shards downloaded and inspected.
* Adapters implemented and unit-smoke-tested.
* Mixed manifest built and validated.
* Per-source per-session parquets written to `data/interim/labeling-corpus/`.
* Re-running `label_corpus.py` would just work on the new manifest.

Cost estimate for the labeling pass when triggered:
* 750 sessions × 2 LLM calls × avg $0.25/call ≈ **$375**.
* Wallclock at concurrency 4 ≈ **5–7 hours**.

## Open questions (flagged, not resolved tonight)

* **Failed-trajectory inclusion**: should the boundary classifier see the
  66k *failed* SWE-agent runs too? They contain rich "agent backtracked"
  signal that may be exactly what the boundary classifier needs to learn.
  Defer to v3 once we see v2 boundary-class distribution.
* **Out-of-distribution risk**: SWE-agent's tool vocabulary (`open`, `goto`,
  `bash`) differs from Copilot CLI's. Either map both to a unified abstract
  vocabulary at the feature layer or let the classifier learn source-specific
  embeddings. Currently planned: keep the raw tool names and let the
  embedding handle it; add `source` as a feature.
* **Trajectory text-vs-code ratio**: SWE-agent observations include raw
  shell output, which can be very long. Canonical-view rendering's
  mid-elision already handles this, but elision ratios will spike on this
  source. Will need to look at the elision counts in v2 and consider a
  per-source elision budget.
